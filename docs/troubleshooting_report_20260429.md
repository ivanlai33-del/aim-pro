## 🛑 1. 問題現象 (Symptoms)
在整合過程中，系統遭遇了連續的技術阻礙：
1. **401 Unauthorized**: 金鑰完全不被認可。
2. **403 Forbidden**: 金鑰被 Google 標記為 Leaked（洩漏）或 Turnstile 驗證攔截。
3. **404 Not Found**: 系統回報找不到 `gemini-2.0-flash` 或 `gemini-1.5-flash` 模型。
4. **429 Too Many Requests**: 免費版帳號配額為 0，導致請求被拒絕。

## 🔍 2. 深度診斷 (Root Cause Analysis)

### A. 環境變數的快取陷阱
*   **發現**：Next.js 的開發伺服器會快取 `.env.local`。即便在檔案中修改了金鑰，程式碼內部 `process.env` 拿到的可能還是舊的「已失效金鑰」。
*   **對策**：執行「毀滅級重啟」：`rm -rf .next` + `pkill node`。

### B. 金鑰身分與權限錯位
*   **發現**：Google 對「新付費帳號」的模型權限有特殊限制。雖然帳號已轉付費，但 Google 依然將某些模型（如 2.0-flash）標記為「不對新用戶開放」。
*   **對策**：使用診斷代碼列出該 API Key 的 **「可用模型清單 (Available Models List)」**，找尋真正存在的標識符。

### C. SDK 的預設行為干擾
*   **發現**：Google AI SDK 會自動切換 `v1` 與 `v1beta` 版本，並自動在模型名稱前補上 `models/`。這種自動化在特定權限受限的專案下會導致 404 錯誤。
*   **對策**：捨棄 SDK，改用 **原生 `fetch`** 呼叫 `v1beta` 端點，精確控制請求路徑。

## ✅ 3. 最終解決方案 (Final Implementation)

### 🚀 技術堆疊決策
*   **模型路徑**：使用 `v1beta/models/gemini-flash-latest` (已被驗證為該帳號最穩定的路徑)。
*   **通訊方式**：原生 `fetch` 呼叫。
*   **認證機制**：環境變數動態注入，支援使用者在網頁介面動態切換（優先順序：使用者金鑰 > 系統金鑰）。

## 🧠 4. 核心經驗價值 (Key Takeaways)

> [!IMPORTANT]
> **給未來的開發建議：**
> 1. **不要在對話中貼出金鑰**：AI 系統可能會自動撤銷被公開的金鑰，導致 403。
> 2. **優先確認可用模型**：當遇到 404 時，立刻執行 `ListModels` 請求，不要盲目猜測模型名稱。
> 3. **底層才是真理**：SDK 出錯時，退回到最原始的 HTTP Request 是唯一的排錯生路。
> 4. **帳號升級與延遲**：Google 帳號升級付費後，API 的權限同步可能存在數分鐘至一小時的「配額釋放期」。
